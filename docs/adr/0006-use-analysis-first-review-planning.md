# Use Analysis-First Review Planning

Local Review will plan a review session before spending expensive model work: it segments the change set, assigns context and exploration budgets, and then runs review passes where the reviewer model may request focused repository exploration as needed. This keeps full change set coverage while avoiding the lower-quality and higher-cost pattern of sending broad repository context to the model by default.

**Considered Options**

- Send as much repository and diff context as possible to each review pass.
- Use simple fixed-size diff chunking with no risk-aware planning.
- Plan review segments and budgets first, then let the reviewer model request bounded exploration during review.

**Consequences**

The review engine needs explicit planning concepts such as review segments, file review segments, hunk review partitions, profile-specific review passes, large change sets, session overviews, context budgets, context bundles, repository exploration scope, ignore boundaries, sensitive file guardrails, repository exploration budgets, exploration rounds, exploration guardrails, repository exploration logs, and review pass statuses. The first implementation should keep initial planning simple and use file review segments as the primary segmentation strategy, then run one profile-specific review pass for each selected profile that applies to that file. If one changed file exceeds a single pass budget, the file review segment may be split into hunk review partitions while preserving the file as the coverage unit. This costs more than combining all criteria into one pass, but keeps feedback traceable to the profile that produced it and avoids mixing incompatible review criteria. For a large change set, defined in v1 as more than 10 changed files or more than 800 modified lines, an internal session overview pass may produce a concise map of the change set to orient later passes, but it must not produce review feedback. These overview thresholds are fixed and not user-configurable in v1. The reviewer model starts from one changed file and one selected profile, then requests bounded file reading, code search, lightweight parsing, generated-file detection, or symbol/import context only when needed. Repository exploration scope covers any non-ignored file in the reviewed repository, not only the change set or nearby references; it respects `.gitignore` and other configured ignore files, and sensitive files must not enter model context. v1 will not support manual overrides for ignored or sensitive files. The model may request multiple exploration rounds, but hard guardrails must prevent loops and context degradation through limits on requests, inspected files, bytes or tokens added, and elapsed time. Review pass status is an observable output contract, not a claim about hidden model reasoning: a pass can complete, complete with limited context, or remain incomplete based on exploration usage, missing context, and evidence. v1 will not run a separate model-based verification pass or automatic feedback deduplication; quality checks are deterministic, and semantic quality relies on structured pass fields plus human curation. It should not run tests, linters, typecheck, scanners, builds, or project validation commands in v1.
